Data extraction

IFC Toolkit can help extract model data into structured schedules or spreadsheets for element review, property review, COBie support, classification review, asset data checks and QA reporting.

Preparing good IFC files

  • Use agreed project naming standards.
  • Export correct IFC entities where possible.
  • Avoid unnecessary proxy objects.
  • Populate required properties before export.
  • Use consistent classification references.
  • Check spatial structure before issuing.
  • Coordinate IFC export settings across disciplines.

COBie QA / QC checker

The native COBie QA / QC tool validates uploaded .xlsx workbooks against configurable COBie worksheet definitions, rule packs, required fields, PickLists, cross-sheet references and practical asset data quality checks.

Preparing a workbook

Use a COBie workbook with standard worksheets such as Contact, Facility, Floor, Space, Type, Component, System, Attribute and PickLists. Keep row 1 as column headings and avoid merged cells or formulas in handover data areas.

Rule packs

Basic COBie, Strict COBie, Asset handover, ISO 19650 aligned and IFC Toolkit recommended rule packs adjust severity, required sheets, required fields, column order sensitivity, placeholder handling and ExternalIdentifier expectations.

Interpreting results

Errors usually block reliable exchange, warnings identify practical quality risks, and info items highlight review points such as project-specific extra columns. The issue table shows sheet, row, column, rule, actual value and recommended correction.

Exports and temporary files

After validation you can download an Excel QA report, CSV issue list and, where enabled, a marked-up workbook with highlighted cells and comments. Uploaded COBie workbooks and generated reports are temporary session files and are not retained permanently.

Limitations

The checker supports COBie validation and QA workflows, but it does not guarantee project, contractual, client, ISO 19650, security, or asset information compliance by itself. Project teams remain responsible for checking against their own AIR, EIR, BEP, exchange information requirements, security protocols and contractual obligations.
